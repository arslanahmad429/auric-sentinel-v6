AURIC SENTINEL
DOCUMENT 04: SIGNAL ENGINE
================================================================

1. PURPOSE

The Signal Engine is the module responsible for identifying specific,
actionable entry trigger conditions on the execution timeframe. It
consumes the directional bias from the EMA Trend Engine and the
structural context from the Market Structure module, and determines the
precise bar and condition at which a candidate trade signal should be
raised. This module does not decide whether a signal survives filtering
(that responsibility belongs to the Filter Engine); it decides where
and when a candidate signal originates.

2. WHY THIS MODULE EXISTS

Trend and structure context alone do not tell a trader exactly when to
act. The Signal Engine exists to translate accumulated context into a
precise, repeatable, rule-based trigger point, removing ambiguity about
"is this the moment" that a trader would otherwise have to judge
manually.

3. FUNCTIONAL REQUIREMENTS

- The module must define at least one clear, rule-based entry trigger
  pattern on the execution timeframe (for example, a pullback into a
  defined zone followed by a confirming reversal candle, or a
  confirmed micro structure shift within an aligned higher timeframe
  order block).
- Each candidate signal must only be considered valid if it aligns with
  both the EMA Trend Engine's directional bias and the Market
  Structure module's structural alignment flag, unless the system
  design explicitly defines a separate counter-trend signal category
  with its own distinct rules.
- The module must maintain a signal lifecycle: candidate identified,
  candidate confirmed (or invalidated), signal issued, signal expired
  or superseded by a new signal.
- The module must prevent duplicate or redundant signals from firing
  repeatedly for what is effectively the same setup (signal memory).
- The module must clearly distinguish between a "long" and "short"
  signal type, and must support disabling either direction
  independently if the trader wants a directional-only mode.

4. NON-FUNCTIONAL REQUIREMENTS

- Signal confirmation must only occur on confirmed, closed bars. No
  signal may be issued based on a still-forming bar's price action.
- The module must be deterministic: given the same historical data, the
  same signal must be produced at the same bar every time the script is
  recalculated, with no randomness or session-dependent variation.
- Signal generation logic must be computationally efficient enough to
  support multi-symbol scanning (Document 09).

5. USER CONFIGURABLE SETTINGS

- Selection between available entry trigger patterns, if more than one
  is implemented.
- Minimum required structural alignment strictness (for example,
  requiring both trend and structure agreement versus allowing trend-
  only alignment).
- Signal memory duration/cooldown (how long after a signal fires before
  a new candidate in the same direction and same zone can be
  considered).
- Toggle for enabling/disabling long signals, short signals, or both.
- Toggle for an optional counter-trend/reversal signal mode, clearly
  separated from the primary trend-aligned mode.

6. INPUT DESCRIPTION

Primary inputs are: the directional bias and strength from the EMA
Trend Engine, the structural alignment flag and active zone information
from the Market Structure module, and execution timeframe price action
used to detect the precise trigger pattern.

7. OUTPUT DESCRIPTION

The module's output, consumed by downstream modules, consists of:
- A signal event (long or short) with the bar at which it was
  confirmed.
- A reference to the structural zone or condition that justified the
  signal (for traceability and dashboard explanation).
- A signal confidence descriptor derived from how strongly trend and
  structure aligned at the time of the signal.
- A signal status (active, expired, superseded, invalidated).

8. VISUAL BEHAVIOUR

Confirmed signals should be marked clearly on the chart at the bar of
confirmation, using a marker style and color that distinguishes long
from short signals at a glance. Once marked, a signal's chart marker
must never move or disappear retroactively; if a signal is later
invalidated by subsequent logic, that must be reflected as a new,
separate visual state rather than erasing the original marker.

9. SIGNAL BEHAVIOUR

A signal should only be issued once per qualifying setup. After a
signal is issued, the module must enter a cooldown or memory state that
prevents an essentially identical setup from re-triggering redundantly
until price action meaningfully changes (for example, until a new
structural event occurs or the prior signal is meaningfully invalidated
or completed).

10. TRADING PHILOSOPHY BEHIND THIS MODULE

The Signal Engine embodies a "confluence before trigger" philosophy:
no single condition alone is sufficient to issue a signal. Trend
context, structural context, and a precise execution timeframe trigger
pattern must all align. This reduces signal frequency in exchange for
higher conviction per signal, consistent with an institutional,
patience-oriented trading style rather than a high-frequency scalping
style.

11. DECISION PROCESS

On each confirmed execution timeframe bar:
1. Check current EMA Trend Engine bias and Market Structure alignment
   flag.
2. If both indicate agreement (or the minimum configured alignment
   strictness is met), scan for the defined execution timeframe trigger
   pattern.
3. If the trigger pattern is confirmed on the current closed bar, and
   no active cooldown applies, issue a new signal.
4. Record the signal's justifying context (zone reference, trend
   strength, structural event) for downstream explanation and
   dashboard display.
5. Update signal memory/cooldown state.

12. EDGE CASES

- Trend and structure briefly disagree right at the moment a trigger
  pattern would otherwise fire: module must suppress the signal rather
  than issuing a low-confidence signal silently, unless counter-trend
  mode is explicitly enabled.
- Rapid, choppy price action producing multiple near-simultaneous
  trigger patterns: signal memory/cooldown must prevent signal spam.
- Trigger pattern forms but higher timeframe context is "undetermined"
  (per Document 02's edge case handling): module must not issue a
  signal when upstream context is undetermined.
- Symbol with very low liquidity or erratic price behavior producing
  frequent false trigger patterns: this should be acknowledged as a
  known limitation rather than solved through overfitting the trigger
  pattern.

13. PERFORMANCE EXPECTATIONS

Signal frequency should be low enough to represent genuinely selective,
high-conviction setups rather than frequent, low-quality triggers. The
exact acceptable frequency should be validated through historical
review during development rather than assumed in advance.

14. FUTURE EXPANSION IDEAS

- Multiple selectable trigger pattern "profiles" (for example, a
  conservative profile requiring stricter alignment versus an
  aggressive profile requiring looser alignment).
- Signal confidence scoring exposed numerically for advanced users, in
  addition to the plain-language descriptor.

15. DEVELOPER NOTES

- Keep the signal lifecycle state machine explicit and well-commented
  internally so that "why did this signal not fire" can always be
  traced back to a specific unmet condition.
- Avoid embedding filter logic (session, volatility) directly inside
  the Signal Engine; that responsibility belongs to the Filter Engine
  (Document 05) and should remain a separate, later-stage check.

16. ACCEPTANCE CRITERIA

- No signal is ever issued without both required upstream context
  conditions being satisfied (unless counter-trend mode is explicitly
  and separately enabled).
- No duplicate signals fire for the same underlying setup within the
  cooldown period.
- Signal markers are stable and never repaint on historical review.

17. TESTING CHECKLIST

- Verify no signals appear during "undetermined" trend states.
- Verify signal memory prevents rapid duplicate signals during choppy
  conditions.
- Verify long and short signal toggles function independently.
- Verify signal justification data (zone reference, confidence
  descriptor) is accurate and traceable for a sample of historical
  signals.

18. KNOWN LIMITATIONS

- Requiring multi-layer confluence necessarily reduces signal
  frequency; this is an intentional design tradeoff favoring quality
  over quantity, and should not be treated as a defect to be "fixed" by
  loosening core alignment requirements.

19. INTEGRATION REQUIREMENTS

- Requires finalized output from both the EMA Trend Engine (Document
  02) and Market Structure module (Document 03).
- Output feeds directly into the Filter Engine (Document 05), Risk
  Management (Document 08), Dashboard (Document 06), and Alerts
  (Document 07).

20. COMMON IMPLEMENTATION MISTAKES TO AVOID

- Allowing the trigger pattern to evaluate against unconfirmed bars.
- Mixing filtering logic into signal generation logic, blurring module
  responsibility boundaries.
- Failing to implement signal memory, resulting in redundant signal
  spam during ranging or choppy conditions.

21. QUALITY ASSURANCE CHECKLIST

- Every issued signal can be traced to explicit trend, structure, and
  trigger pattern justification.
- Signal frequency, reviewed historically, reflects a selective rather
  than noisy character.
- No retroactive changes to previously issued signals during historical
  scrolling.

22. THE COMPLETE SIGNAL DECISION HIERARCHY

This section describes, in full detail, the complete multi-stage
decision process the Signal Engine must follow before any signal can be
approved and passed to the Filter Engine (Document 05). This hierarchy
is the central logic of the entire AURIC SENTINEL system, and every
other module exists to feed one stage of this process. As with all
sections in this documentation set, this describes required behaviour
and expected reasoning flow only; no implementation method, formula, or
code is specified.

The decision hierarchy proceeds through the following ordered stages
for every candidate setup:

Stage 1 - Trend
Stage 2 - Market Structure
Stage 3 - EMA Confirmation
Stage 4 - Price Action
Stage 5 - Momentum
Stage 6 - Filters
Stage 7 - Risk Validation
Stage 8 - Final BUY/SELL Decision

Each stage acts as a gate: a candidate setup must pass the current
stage's requirements before being evaluated against the next stage. A
failure at any stage results in the candidate setup being downgraded or
rejected, per the specific rules of that stage, rather than being
silently discarded without explanation.

22.1 Stage 1 - Trend

The candidate setup must first be evaluated against the EMA Trend
Engine's current bias, strength, and Trend Confidence Rating (Document
02, Sections 3 and 22.19). A candidate aligned with a high-confidence
trend proceeds with full standing. A candidate aligned with a
lower-confidence or transitional trend state may still proceed, but
must be flagged for reduced Signal Confidence (Section 22.10) at later
stages. A candidate directly opposed to a high-confidence trend must be
rejected at this stage unless the system's counter-trend mode
(Document 04, Section 5) is explicitly enabled, in which case it
proceeds through a separate, more demanding counter-trend evaluation
path rather than the standard path.

22.2 Stage 2 - Market Structure

The candidate setup must next be evaluated against the Market
Structure module's current alignment flag, Market Phase, and Structure
Strength Rating (Document 03, Sections 7 and 22.9-22.10). A candidate
occurring during a clearly Trending phase with confirmed, high-strength
structural alignment proceeds with full standing. A candidate occurring
during a Ranging, Transitional, or Accumulation/Distribution phase, or
against unconfirmed/provisional structural events (Document 03, Section
22.12), must be flagged for reduced confidence rather than rejected
outright, unless the structural signal is in direct, confirmed conflict
with the candidate's direction, in which case it is rejected at this
stage.

22.3 Stage 3 - EMA Confirmation

The candidate setup must be evaluated against the specific EMA
behavioural state described in Document 02, Section 22, including
Healthy versus Weak Trend classification, presence of Ribbon Expansion
or Compression, Acceleration or Deceleration flags, and Trend
Exhaustion status. A candidate occurring during a Healthy Trend with
Expansion and Acceleration proceeds at full standing. A candidate
occurring during Early Trend Weakness or Trend Exhaustion must be
flagged for reduced confidence. A candidate occurring after confirmed
Trend Failure (Document 02, Section 22.15) in the candidate's own
direction must be rejected at this stage.

22.4 Stage 4 - Price Action

The candidate setup must exhibit the specific execution-timeframe
trigger pattern defined in Document 04, Section 3, evaluated with
reference to Pullback Detection, EMA Rejection Detection (Document 02,
Sections 22.8-22.9), and Impulse/Correction Move classification
(Document 03, Sections 22.6-22.7). The price action stage exists to
confirm that the precise, granular price behavior at the point of
candidate formation is consistent with a genuine continuation or
reversal pattern rather than incidental, directionless movement. A
candidate lacking clear, decisive price action confirmation must be
flagged for reduced confidence rather than automatically rejected,
since later stages may still justify a lower-conviction signal
category.

22.5 Stage 5 - Momentum

The candidate setup must be evaluated against available momentum
context on the execution timeframe (for example, whether recent price
movement shows accelerating or decelerating follow-through consistent
with the candidate's direction, conceptually parallel to the
Acceleration/Deceleration concepts defined for the higher timeframe in
Document 02, Sections 22.6-22.7, but assessed at the execution
timeframe). A candidate with confirming momentum proceeds at full
standing. A candidate with contradicting or absent momentum
confirmation must be flagged for reduced confidence.

22.6 Stage 6 - Filters

The candidate setup, having passed or been appropriately flagged
through Stages 1-5, is passed to the Filter Engine (Document 05) for
session, volatility, and noise evaluation. This stage is external to
the Signal Engine's own internal logic but is included in this
hierarchy description for completeness, since a candidate rejected here
never proceeds regardless of how favorably it scored in prior stages.

22.7 Stage 7 - Risk Validation

Before final approval, the candidate setup must be evaluated against
the Risk Management module's safeguards (Document 08, Section 12),
including minimum and maximum reasonable stop-distance checks. A
candidate that would produce an invalid or unreasonable risk framing
(for example, an unreasonably tight or unreasonably wide stop) must be
rejected at this stage regardless of how favorably it scored in all
prior stages, since a technically excellent setup with unworkable risk
framing provides no practical trading value.

22.8 Stage 8 - Final BUY/SELL Decision

Only after successfully passing all prior stages (with any accumulated
confidence reductions carried forward) does the candidate setup become
a final, approved BUY or SELL decision. This final decision carries
forward the complete accumulated context from every stage: which stages
were passed at full standing, which stages contributed a confidence
reduction, and the specific justification recorded at each stage. This
complete context must be available to the Dashboard (Document 06) and
Alerts (Document 07), not merely the final direction and price.

22.9 Signal Quality

Signal Quality is a qualitative descriptor summarizing how cleanly a
signal passed through the decision hierarchy above. A signal that
passed every stage at full standing is of the highest quality tier. A
signal that passed but accumulated one or more confidence reductions
along the way is of a lower quality tier, clearly distinguished from
the highest tier so the trader can weight their own discretionary
judgment accordingly.

22.10 Signal Priority

When multiple candidate setups qualify in close proximity (whether on
the same instrument at different structural levels, or across multiple
instruments when Scanner Optimization, Document 09, is enabled), the
system must be able to rank them by priority, derived from Signal
Quality, Signal Confidence (Section 22.11), and Structure Strength
Rating. This priority ranking exists to help the trader decide which
setup deserves primary attention when faced with more than one
qualifying opportunity at once.

22.11 Signal Confidence

Signal Confidence is the accumulated numerical or plain-language
confidence measure built up across all eight decision hierarchy stages,
synthesizing the Trend Confidence Rating, Structure Strength Rating,
EMA behavioural state, price action clarity, and momentum confirmation
into a single unified score or tier. This is the primary confidence
measure displayed to the trader and referenced by Alerts (Document 07)
and Risk Management (Document 08) for any confidence-sensitive
behaviour.

22.12 Signal Rejection

Signal Rejection occurs when a candidate setup fails a hard requirement
at any stage (direct conflict with high-confidence trend, direct
conflict with confirmed structure, confirmed Trend Failure, invalid
risk framing, or Filter Engine rejection). Every rejection must be
traceable to the specific stage and specific reason that caused it,
consistent with the traceability principle already established for
Filter Engine rejections (Document 05, Section 7).

22.13 Signal Invalidation

Signal Invalidation is distinct from Signal Rejection: it describes a
previously approved, active signal that subsequently fails to remain
valid due to changing market conditions (for example, a confirmed
Structure Failure occurring after the signal was issued, or price
decisively closing back through the structural zone that justified the
signal). Invalidation must be reported as a new, clearly labeled state
change on the existing signal, consistent with the non-retroactive
state-change principle established in Document 08, Section 9, and must
trigger an appropriate Alerts notification if enabled.

22.14 Entry Timing

The module must distinguish between the theoretical moment a candidate
setup first became structurally valid and the specific, actionable bar
at which the full decision hierarchy was satisfied and the signal was
actually issued. This distinction exists so the trader understands that
"the setup started forming" is different from "the setup is now
actionable," reducing confusion about apparent discrepancies between
chart structure and signal timing.

22.15 Late Entries

A Late Entry describes a condition where a trader attempts to act on a
signal a meaningful number of bars after it was originally issued, such
that the original risk framing (Document 08) may no longer be
favorable (for example, price has already moved a significant portion
of the way to target). The module should support flagging signals as
"aged" once a defined threshold of bars or price movement since
issuance has elapsed, so the Dashboard can visually communicate reduced
suitability for fresh entry without removing the historical signal
record.

22.16 Missed Entries

A Missed Entry describes a condition where a signal was issued and
subsequently invalidated or completed without the trader having acted
on it. The system is not required to take any corrective action for a
Missed Entry, but should ensure such signals are clearly marked as
closed/historical on the Dashboard (per Document 08, Section 12) so the
trader does not mistakenly treat a missed, closed signal as still
actionable.

22.17 Aggressive Entry

An Aggressive Entry Mode, if enabled by the trader, allows candidate
setups that accumulated one or more confidence reductions during the
decision hierarchy (Sections 22.1-22.5) to still be approved as valid,
lower-quality-tier signals, rather than being held to the same standard
required in the default configuration. This mode must be clearly and
persistently indicated on the Dashboard whenever active, since it
fundamentally changes the selectivity character of the entire system.

22.18 Conservative Entry

A Conservative Entry Mode, if enabled by the trader, raises the
approval standard beyond the default configuration, requiring full
standing (no confidence reductions) at every applicable stage of the
decision hierarchy before a signal is approved. This mode must further
reduce signal frequency in exchange for the highest available
selectivity, consistent with the system's overall "quality over
quantity" philosophy (see Document 11).

22.19 High Probability Entry

A High Probability Entry is a distinct, top-tier signal classification
reserved for candidate setups that pass every stage of the decision
hierarchy at full standing AND additionally satisfy Multi-Timeframe EMA
Alignment (Document 02, Section 22.16) AND occur during a confirmed
Trending Market Phase (Document 03, Section 22.9) with a fully
confirmed (non-provisional) structural event. This classification
represents the rare, highest-conviction condition where every available
layer of the system is in full agreement, and must be visually and
categorically distinguished from standard approved signals on both the
Dashboard and in Alerts.

22.20 Integration Notes for the Decision Hierarchy

The eight-stage hierarchy described in this section does not replace
any requirement described in Sections 1-21 of this document; it
provides the complete, explicit reasoning sequence that those
requirements operate within. Every module referenced in this section
(EMA Trend Engine, Market Structure, Filter Engine, Risk Management)
must expose the specific data points referenced above in a form the
Signal Engine can consume directly, consistent with the module
communication rules established in Document 01, Section 4.
