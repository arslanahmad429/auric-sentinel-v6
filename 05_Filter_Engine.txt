AURIC SENTINEL
DOCUMENT 05: FILTER ENGINE
================================================================

1. PURPOSE

The Filter Engine is the module responsible for evaluating every
candidate signal produced by the Signal Engine against a set of
secondary quality conditions - session timing, volatility conditions,
and general market noise - before a signal is allowed to proceed to
Risk Management, Dashboard, and Alerts. This module acts as a quality
gate, not a signal generator.

2. WHY THIS MODULE EXISTS

Not every technically valid trigger pattern occurs under conditions
favorable to a good outcome. A structurally sound signal that fires
during an illiquid session, or during abnormally low or abnormally
extreme volatility, carries different risk characteristics than the
same pattern occurring during a normal, liquid trading session. This
module exists to separate "the pattern is technically correct" from
"the conditions are favorable for acting on it right now."

3. FUNCTIONAL REQUIREMENTS

- The module must support a session filter that allows the trader to
  define which trading sessions (for example, London, New York, Asian,
  or custom time windows) are considered valid for signal approval,
  expressed and displayed in Pakistan Standard Time (UTC+5).
- The module must support a volatility filter that rejects signals
  occurring during conditions of abnormally low volatility (where
  price movement is unlikely to reach meaningful targets) or abnormally
  extreme volatility (where risk becomes difficult to control),
  relative to a recent historical volatility baseline for the
  instrument.
- The module must support a general noise filter concept that can
  reject signals occurring immediately around known high-impact
  conditions if the trader chooses to enable such a restriction (for
  example, avoiding signals in the minutes immediately surrounding
  major scheduled data releases, if such timing data is available to
  the system).
- The module must clearly report which specific filter (if any)
  rejected a given candidate signal, rather than only reporting a
  binary pass/fail, so the trader can understand why a technically
  valid pattern was not approved.

4. NON-FUNCTIONAL REQUIREMENTS

- Filtering must not introduce any repainting risk; a signal that was
  approved and displayed must never later be silently hidden.
- Filtering must evaluate quickly enough to avoid introducing
  meaningful delay in signal approval on the confirming bar.
- All session time boundaries must be clearly and consistently
  converted to and displayed in Pakistan Standard Time regardless of
  the exchange's native timezone.

5. USER CONFIGURABLE SETTINGS

- Enable/disable toggle for each filter type independently (session,
  volatility, noise).
- Custom session time window definition(s), in PKT.
- Volatility filter sensitivity (how far from the historical baseline
  volatility must be before a signal is rejected).
- Option to treat filters as "hard rejection" (signal fully suppressed)
  versus "soft warning" (signal still shown but visually flagged as
  lower confidence).

6. INPUT DESCRIPTION

Primary inputs are: the candidate signal event and its metadata from
the Signal Engine, current bar's time (for session evaluation), and a
volatility measure derived from recent price action (for example, a
recent-range-based volatility baseline).

7. OUTPUT DESCRIPTION

The module's output, consumed by downstream modules, consists of:
- A final approval or rejection status for each candidate signal.
- If rejected, a specific reason code/descriptor (session, volatility,
  noise) for traceability and dashboard/alert explanation.
- If the "soft warning" mode is enabled, a reduced-confidence flag
  rather than outright rejection.

8. VISUAL BEHAVIOUR

Approved signals proceed to be displayed normally per Document 04 and
Document 06's specifications. Rejected candidate signals should
generally not be displayed as full signals, but the system may
optionally provide a subtle, clearly distinct visual indicator (for
example, a muted marker) for candidates that were rejected only by a
soft filter, so an advanced trader can still see "a pattern almost
qualified here" without confusing it with an actionable signal.

9. SIGNAL BEHAVIOUR

The Filter Engine does not alter the underlying signal's directional
logic; it only gates whether the signal becomes visible/actionable.
A rejected signal must not be issued to Alerts (Document 07) or fully
processed by Risk Management (Document 08).

10. TRADING PHILOSOPHY BEHIND THIS MODULE

The underlying philosophy is that discipline around when to act is as
important as the technical correctness of the setup itself. Many
otherwise valid trading systems underperform not because their core
logic is flawed but because they are applied indiscriminately across
all sessions and volatility regimes. This module operationalizes that
discipline as an explicit, consistent rule set rather than leaving it
to trader memory or willpower in the moment.

11. DECISION PROCESS

For each candidate signal received from the Signal Engine:
1. Evaluate session filter: is the current bar's time within an
   approved session window (in PKT)?
2. Evaluate volatility filter: is current volatility within the
   acceptable band relative to the historical baseline?
3. Evaluate noise filter (if enabled): does any configured noise
   condition apply?
4. If all enabled filters pass, approve the signal and forward it
   downstream.
5. If any enabled filter fails, reject (or soft-flag) the signal and
   record the specific reason.

12. EDGE CASES

- Signal occurs exactly at a session boundary: the module must have a
  clearly defined inclusive/exclusive boundary rule to avoid ambiguous
  behavior.
- Volatility baseline is not yet reliably established (for example,
  very early in a symbol's history): module should default to a
  cautious "insufficient data" rejection rather than approving by
  default.
- Multiple filters would reject the same signal simultaneously: the
  module should report all applicable rejection reasons, or at minimum
  the most relevant one, rather than only ever reporting the first
  filter checked.

13. PERFORMANCE EXPECTATIONS

Filtering should not meaningfully reduce the responsiveness of the
system; approval or rejection should be resolved on the same confirmed
bar as the original candidate signal.

14. FUTURE EXPANSION IDEAS

- Configurable "quality score" that blends multiple filter dimensions
  into a single confidence adjustment rather than strictly binary
  pass/fail per filter.
- Historical statistics tracking of how often each filter rejects
  signals, to help the trader tune filter sensitivity over time.

15. DEVELOPER NOTES

- Keep each filter as an independently toggleable, independently
  testable unit so that the trader can isolate which filter is
  responsible for any given rejection during testing.
- Ensure the session filter's timezone conversion logic is implemented
  once and reused consistently everywhere in the system that displays
  time, to avoid inconsistent time displays between modules.

16. ACCEPTANCE CRITERIA

- Every rejected signal has a traceable, specific rejection reason.
- Session windows behave consistently and correctly in PKT across
  daylight-saving or exchange timezone differences, if applicable to
  the instrument.
- Volatility filter meaningfully distinguishes abnormal conditions from
  normal conditions in historical review.

17. TESTING CHECKLIST

- Test signals occurring inside and outside each defined session
  window.
- Test signals occurring during simulated abnormally low and abnormally
  high volatility.
- Test with all filters disabled to confirm all technically valid
  Signal Engine output passes through unmodified.
- Test with all filters enabled simultaneously to confirm correct
  combined behavior.

18. KNOWN LIMITATIONS

- Volatility filtering based on historical baselines is inherently
  reactive and may not perfectly anticipate sudden regime changes.

19. INTEGRATION REQUIREMENTS

- Requires candidate signal input from the Signal Engine (Document 04).
- Approved output feeds Risk Management (Document 08), Dashboard
  (Document 06), and Alerts (Document 07).

20. COMMON IMPLEMENTATION MISTAKES TO AVOID

- Silently dropping signals without recording a rejection reason,
  making the system's behavior feel opaque to the trader.
- Inconsistent timezone handling between this module and other modules
  that display time.
- Applying filters so strictly by default that the system approves an
  impractically small number of signals without the trader
  understanding why.

21. QUALITY ASSURANCE CHECKLIST

- Rejection reasons are accurate and specific in manual spot checks.
- Session and volatility filters behave predictably at their boundary
  conditions.
- No signal that should have been rejected is mistakenly approved, and
  vice versa, across a representative historical test sample.
